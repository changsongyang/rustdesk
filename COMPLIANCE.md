## 2. 安全策略

### 2.1 秘密管理

**策略**: 所有敏感信息必须通过 GitHub Secrets 或环境变量管理，禁止硬编码。

**管理的敏感信息**:

| 类型 | 描述 | 存储位置 |
|------|------|----------|
| 签名证书 | Android keystore、macOS P12 证书 | GitHub Secrets |
| API 密钥 | 签名服务、通知服务密钥 | GitHub Secrets |
| 服务器配置 | 中继服务器地址、API 端点 | GitHub Secrets |
| 凭据 | 密码、令牌等认证信息 | GitHub Secrets |

**安全措施**:
- Secrets 使用 AES-256 加密存储
- 最小权限原则：仅授权必要的工作流访问特定 secrets
- 定期轮换敏感凭据（建议每 90 天）

### 2.2 依赖安全

**策略**: 定期审查和更新项目依赖，防止已知漏洞。

**审查机制**:
1. **依赖审查**：使用 GitHub Dependency Review 检查所有 PR 的依赖变更
2. **安全审计**：使用 cargo-deny 进行依赖漏洞扫描
3. **许可证合规**：仅允许开源许可证（MIT、Apache-2.0、BSD 等）

**允许的许可证列表**:
- MIT License
- Apache License 2.0
- BSD-2-Clause
- BSD-3-Clause
- CC-BY-4.0
- ISC License
- MPL-2.0
- Unlicense
- Zlib License

### 2.3 代码签名

**策略**: 所有发布的二进制文件必须经过代码签名。

| 平台 | 签名方式 |
|------|----------|
| Windows | Authenticode 签名 |
| macOS | Apple Developer ID 签名 + Notarization |
| iOS | Apple Developer 签名 |
| Android | APK 签名（v1 + v2 + v3） |

## 3. 数据保护措施

### 3.1 端到端加密

RustDesk 采用端到端加密确保远程会话的安全性：

- **加密算法**: AES-256-GCM
- **密钥交换**: 椭圆曲线 Diffie-Hellman (ECDH)
- **证书验证**: 自签名证书或 CA 颁发证书

### 3.2 剪贴板安全

- 剪贴板数据在传输过程中加密
- 支持选择性剪贴板同步
- 敏感剪贴板内容（如密码）在会话结束后自动清除

### 3.3 屏幕共享权限

- 用户必须明确授权屏幕共享
- 支持窗口级别的屏幕共享（而非整个屏幕）
- 屏幕共享状态在系统托盘可见指示

## 4. 访问控制机制

### 4.1 GitHub 仓库权限

**角色定义**:

| 角色 | 权限 | 数量限制 |
|------|------|----------|
| 仓库管理员 | 完全控制 | ≤ 3 人 |
| 代码所有者 | 代码审查、合并 PR | ≤ 5 人 |
| 贡献者 | 提交 PR | 按需 |
| 外部贡献者 | 提交 PR（需审查） | 不限 |

**权限原则**:
- 最小权限原则：仅授予完成工作所需的最低权限
- 定期审查：每季度审查权限分配
- 即时撤销：离职或角色变更时立即撤销访问权限

### 4.2 CI/CD 权限

- **GitHub Token**: 使用 `GITHUB_TOKEN` 进行 API 访问，限制为仓库级别权限
- **Runner 隔离**: 使用 GitHub 托管的 runner，避免自托管环境的安全风险
- **工作流限制**: 工作流仅能访问其所需的 secrets

## 6. 合规性检查流程

### 6.1 预合并检查

在 PR 合并前必须通过以下检查：

1. **依赖审查**: 检查新增依赖是否包含已知漏洞
2. **代码审查**: 至少 1 位代码所有者批准
3. **自动化测试**: 所有测试通过
4. **安全扫描**: TruffleHog 扫描无高危发现

### 6.2 CI/CD 安全门禁

工作流架构：

```
┌──────────────────────────────────┐
│   compliance-check (安全门禁)    │
│  ├── Dependency Review          │
│  └── Secret Scanning            │
└──────────────┬───────────────────┘
               ▼
┌──────────────────────────────────┐
│    generate-bridge (桥接生成)    │
└──────────────┬───────────────────┘
               ▼
┌──────────────────────────────────┐
│     各平台构建作业               │
│  ├── Windows                    │
│  ├── macOS                      │
│  ├── Linux                      │
│  ├── Android                    │
│  └── iOS                        │
```

## 8. 应急响应流程

### 8.1 安全事件分类

| 级别 | 描述 | 响应时间 |
|------|------|----------|
| **P0** | 严重安全漏洞，影响用户数据 | 立即 |
| **P1** | 高危漏洞，可能导致服务中断 | 4 小时内 |
| **P2** | 中危漏洞，需要计划修复 | 24 小时内 |
| **P3** | 低危漏洞，例行修复 | 7 天内 |

### 8.2 响应步骤

1. **发现**: 通过安全扫描或用户报告发现漏洞
2. **评估**: 安全团队评估漏洞严重性和影响范围
3. **修复**: 开发团队实施修复方案
4. **验证**: 验证修复有效性
5. **通知**: 向受影响用户发送安全公告
6. **复盘**: 分析根因，更新安全策略

### 8.3 联系人

| 角色 | 职责 | 联系方式 |
|------|------|----------|
| 安全负责人 | 安全事件协调 | security@rustdesk.com |
| 紧急联系人 | 7x24 响应 | emergency@rustdesk.com |
